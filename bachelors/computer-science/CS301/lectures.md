## Lecture 1: The Distributed Systems Landscape — Motivations, Definitions, and Fundamental Challenges

Distributed systems are everywhere in 2040, yet they remain one of the most intellectually challenging domains of computer science. A distributed system is, in the classic formulation of Leslie Lamport, "one in which the failure of a computer you didn't even know existed can render your own computer unusable." More formally, a distributed system is a collection of independent computers that appear to the user as a single coherent system. The computers — called *nodes* — communicate by exchanging messages over a network, each node has its own memory and executes its own programs, and there is no shared clock or shared memory that would allow global synchronisation. It is precisely the absence of these things — the **no shared clock**, **no shared memory**, **no global state** constraints — that makes distributed systems both powerful and difficult.

The motivations for building distributed systems are compelling and have only intensified since the 2020s. **Resource sharing**: distributed systems allow geographically separated users and applications to share data, computation, and specialised hardware. **Scalability**: by adding more nodes, a distributed system can handle more load — in principle, the system can scale horizontally to arbitrary size. **Availability and fault tolerance**: if one node fails, others can take over — the system as a whole remains operational even when individual components fail, provided that the system is designed for redundancy. **Performance**: by placing computation and data close to where they are needed — the principle of *edge computing* — distributed systems can reduce latency and improve throughput. **Cost efficiency**: a distributed system built from many inexpensive commodity nodes can be more cost-effective than a single, ultra-reliable mainframe.

The fundamental challenges that distributed systems must overcome derive from the physics and topology of the network. **Latency**: the time for a message to travel between nodes is not zero and is not predictable — it varies with distance, congestion, and routing. **Partial failure**: in a distributed system, one component can fail while others continue to operate — and the system must detect, isolate, and recover from such failures without stopping the whole. The hardest part is that partial failure is *incomplete information*: a node that has not responded for one second may have failed, or may be slow, or the message may have been lost, or the network path may have been temporarily partitioned. **No global clock**: each node has its own clock, and these clocks drift relative to each other — there is no way for two nodes to agree on the precise time at which an event occurred without additional protocols. **Concurrency**: without shared memory, nodes must coordinate through message passing, and the classic problems of concurrency (race conditions, deadlock, livelock) reappear in a distributed setting with the added difficulty that no node has a complete picture of the system state.

The 2040 landscape of distributed systems is shaped by several transformative trends. **Global-scale cloud infrastructure**: services like AWS, Azure, and Google Cloud operate hundreds of data centres worldwide, each containing hundreds of thousands of servers, all coordinated as a single distributed system. **Edge computing**: computation and data storage are pushed to the edge of the network — to base stations, routers, IoT gateways, and even the devices themselves — to reduce latency and bandwidth usage. **Blockchain and decentralised systems**: permissionless distributed systems that operate without any central authority, using consensus protocols to maintain a shared ledger across thousands of nodes, many of which may be Byzantine (arbitrarily malicious). **Decentralised AI inference**: large language models and other neural architectures are increasingly served by distributed inference systems that split model layers across many GPUs, coordinate their execution, and aggregate results under strict latency budgets. The University of Yggdrasil's own Yggdrasil Distributed Computing Laboratory — a 32-node Raspberry Pi 5 cluster running custom fault-tolerance experiments — exemplifies the kind of hands-on experimentation that is essential for understanding these systems.

## Lecture 2: Communication Models and Remote Procedure Call — From Message Passing to Modern RPC Frameworks

At the heart of every distributed system is the communication between nodes. The simplest model is **message passing**: one node sends a message to another, and the receiving node processes it. Message passing comes in two flavours: **synchronous** (the sender blocks until the receiver acknowledges receipt and optionally returns a response) and **asynchronous** (the sender sends the message and continues immediately, with the response arriving later, possibly on a different thread or callback). Synchronous message passing is simpler to program but wastes time while waiting; asynchronous message passing is more efficient but requires more complex programming models (callbacks, promises, futures, event loops).

The **Remote Procedure Call (RPC)** abstraction — first formulated by Bruce Jay Nelson in his 1984 PhD thesis — attempts to make distributed communication look like a local procedure call. The programmer writes code that calls a function; the function's body executes on a remote node; the result is returned to the caller as if the call were local. The RPC system handles the details: marshalling (serialising the arguments into a network message), unmarshalling (deserialising them on the remote end), transmitting the message, handling errors, and returning the result. The illusion is powerful, but it is — and was always known to be — a leaky abstraction. **Latency**: a local procedure call takes nanoseconds; an RPC takes milliseconds — six orders of magnitude slower. **Partial failure**: a local procedure call either succeeds or fails with a crash; an RPC can succeed on the server but the response can be lost — the caller cannot know whether the operation was performed. **Memory semantics**: local procedures share memory; RPCs must copy data across the network, which changes the semantics of pointers, large data structures, and side effects.

Modern RPC frameworks — gRPC (Google), Apache Thrift, Cap'n Proto, and the emerging WebAssembly-based WIT/RPC standard — address these challenges through several architectural innovations. **Protocol buffers** (protobufs): a language-neutral, platform-neutral mechanism for serialising structured data, with schemas defined in `.proto` files and compilers that generate client and server stubs in virtually every programming language. The protobuf encoding is compact, deterministic, and backward-compatible, making it the standard for gRPC and many other systems. **Streaming**: unlike traditional RPC (one request, one response), gRPC supports four kinds of streaming — unary (single request, single response), server-streaming (single request, stream of responses), client-streaming (stream of requests, single response), and bidirectional streaming (both sides send and receive streams) — enabling real-time applications like video feeds, sensor data collection, and collaborative editing. **Service meshes**: in 2040, most production RPC traffic is not handled directly by application code but is routed through a sidecar proxy — typically a lightweight process deployed alongside each service instance — that handles service discovery, load balancing, retry logic, circuit breaking, observability, and encryption automatically (Istio, Linkerd, Consul Connect). The sidecar proxy pattern decouples the communication infrastructure from the application logic, allowing operations teams to manage RPC behaviour without modifying the codebase.

**Asynchronous messaging** systems — message queues and publish-subscribe systems — offer an alternative model that decouples senders from receivers in both time and space. **Message queues** (RabbitMQ, Apache ActiveMQ, Amazon SQS): a sender places a message on a queue; a receiver — one of potentially many consumers — picks it up and processes it. Messages are durable (survive node failures) and exactly once delivery is provided (at least once delivery in practice, with idempotent consumers). **Publish-subscribe** (Apache Kafka, Google Pub/Sub, NATS): publishers send messages to a topic; subscribers that have registered interest in that topic receive copies of each message. Pub-sub decouples publishers from subscribers completely — they need not know each other's identity, location, or even existence. Kafka, originally developed at LinkedIn and now ubiquitous, has become the de facto standard for distributed event streaming, providing not just pub-sub messaging but also durable storage (messages are persisted to disk and replicated), stream processing (Kafka Streams, ksqlDB), and exactly-once semantics.

**Network partitions** — the split of a distributed system into two or more groups that cannot communicate — are the single most disruptive event a distributed system can face. During a partition, the system must choose: **availability** (continue to respond to requests, risking inconsistency) or **consistency** (refuse to respond until the partition is healed, ensuring all nodes see the same data). This choice is the heart of the CAP theorem, which we examine in Lecture 3.

## Lecture 3: The CAP Theorem and Its Consequences — Consistency, Availability, and Partition Tolerance

The **CAP theorem**, proved by Eric Brewer in 2000 and formally established by Seth Gilbert and Nancy Lynch in 2002, states that a distributed data system can provide at most two of three properties simultaneously: **Consistency** (C), **Availability** (A), and **Partition Tolerance** (P). The theorem is not a vague principle but a mathematical result with a rigorous proof: in any asynchronous distributed system that can experience network partitions, it is impossible to ensure both consistency and availability during a partition.

To understand the theorem precisely, we must define its terms. **Consistency** (in the CAP sense — often called "strong consistency" or "linearisability") means that all nodes see the same data at the same time: after a write completes, every subsequent read returns the value of that write (or a later one). In other words, there is a total order of operations, and every node observes that order consistently. **Availability** means that every request received by a non-failing node must result in a response — not necessarily the most recent data, but *some* response, within a bounded time. The system does not allow indefinite waiting. **Partition Tolerance** means that the system continues to function even if the network is split into two or more disconnected components — messages between nodes in different partitions are lost.

The proof of the CAP theorem is straightforward. Consider a distributed system with two nodes, A and B. A client writes value v₁ to node A. Before A can replicate v₁ to B, the network partitions: A and B can no longer communicate. Now a second client reads from node B. For **consistency**, B must return v₁ — but B does not have v₁ (the partition prevented replication). For **availability**, B must return *some* response — but it cannot return v₁ because it doesn't have it. B is forced to choose: violate consistency (return the old value, or a default) or violate availability (refuse to respond, or return an error). There is no third option. The theorem does not say that consistency and availability are impossible to achieve simultaneously in general — they are impossible to achieve simultaneously *during a partition*. Since partitions are inevitable in any real distributed system (network links fail, routers reboot, cables are cut), the practical consequence is that system designers must choose between CP (consistency + partition tolerance, sacrifice availability during partitions) and AP (availability + partition tolerance, sacrifice consistency during partitions).

The **CA** case — consistency + availability, with no partition tolerance — is theoretically possible only in systems that are not distributed, i.e., systems where partition is impossible (a single-node database, or a system where all nodes are in the same rack with redundant network paths so reliable that partition is considered negligible). In practice, modern distributed systems assume partitions will occur and design for CP or AP.

**CP systems** (e.g., Zookeeper, etcd, most traditional relational databases with synchronous replication): when a partition occurs, the system stops accepting writes on the minority side (the partition that does not have the leader) to prevent inconsistency. The minority side may return errors or refuse to respond — it sacrifices availability. Once the partition is healed, the minority side catches up by replaying the operation log from the leader. The trade-off is that the system is unavailable during the partition for clients connected to the minority side, but consistency is guaranteed.

**AP systems** (e.g., Amazon DynamoDB, Cassandra, Riak, DNS): when a partition occurs, both sides continue to accept writes — consistency is sacrificed. Different clients may see different versions of the data depending on which side they query. Once the partition is healed, the system must reconcile the divergent versions — a process called **conflict resolution**. AP systems typically use **eventual consistency** (the system will eventually converge to a consistent state if writes stop) and rely on **last-writer-wins** (LWW), **version vectors**, or **CRDTs** (Lecture 6) to merge concurrent updates.

Brewer himself, in his 2012 retrospective, argued that the CAP theorem is often misinterpreted. "CAP is not binary," he wrote — "it is a continuum." System designers can trade consistency for availability in fine-grained ways: they can relax consistency for some operations but not others, use quorum-based reads and writes (a *quorum* is a majority of replicas; reading from a quorum and writing to a quorum ensures that at least one node in the read quorum has seen the write — a weaker guarantee than linearisability), or use **hybrid approaches** that provide strong consistency for critical operations (e.g., authentication, payment) and eventual consistency for less critical ones (e.g., user profiles, recommendations). The practical art of distributed systems engineering is not to choose C-A-P in the abstract but to understand the consistency semantics that each operation requires and to design protocols that provide the right level of consistency with the minimum sacrifice of availability.

## Lecture 4: Time, Clocks, and the Ordering of Events

In a single-processor system, time is straightforward: there is a single clock, and the ordering of events is determined by the clock's tick. In a distributed system, there is no such thing. Each node has its own clock, quartz-crystal oscillators drift relative to each other (typically 1–100 parts per million), and even with Network Time Protocol (NTP) synchronisation, the residual skew between two nodes' clocks after synchronisation is on the order of milliseconds on a LAN and tens of milliseconds on the Internet. The fundamental problem is that **two nodes cannot agree on the order of events** by looking at their local clocks — the clocks do not agree, and the uncertainty is large enough to matter for many protocols.

Leslie Lamport's 1978 paper "Time, Clocks, and the Ordering of Events in a Distributed System" is one of the most influential works in the field — it did for distributed systems what Einstein's 1905 paper did for physics: it showed that the notion of a single universal time is not merely impractical but conceptually incoherent in a distributed setting. Lamport's key insight is that we do not need absolute time (what time is it?) to order events — we need only **causal ordering** (did event A happen before event B?). The **happened-before** relation — denoted `A → B` — is defined as follows: (i) if A and B are events on the same node and A occurs before B according to the local clock, then A → B; (ii) if A is the sending of a message and B is the receipt of that message, then A → B; (iii) if A → B and B → C, then A → C (transitivity). Two events are said to be **concurrent** — neither happened before the other, and they are not causally related — if neither A → B nor B → A holds.

Lamport introduced **Lamport clocks** — a logical clock mechanism that assigns a timestamp to each event. Each node maintains a counter (the logical clock), initialised to 0. At every event on a node, the counter is incremented. When a message is sent, its timestamp is the sender's current counter value. When a message is received, the receiver sets its counter to `max(receiver's counter, message's timestamp) + 1`. Lamport clocks have the property that if A → B, then `timestamp(A) < timestamp(B)`. The converse is not true: if `timestamp(A) < timestamp(B)`, we cannot infer A → B — the timestamps alone do not tell us whether the events are causally related or merely concurrent. This is the limitation of Lamport clocks: they provide a *total* order (if timestamps are equal, tie-breaking by node ID gives a total order) but not a *causal* order — we cannot distinguish concurrency from causality.

**Vector clocks**, introduced independently by Friedemann Mattern and Colin Fidge in 1988, solve this problem. A vector clock is a tuple `[c₁, c₂, ..., cₙ]` — one counter for each node in the system. Each node maintains its own vector clock. On each event, the node increments its own entry. When a message is sent, the sender attaches its entire vector clock (all n entries). When a message is received, the receiver updates each entry in its own vector clock: `cᵢ = max(cᵢ_local, cᵢ_message)`. Then it increments its own entry. Two vector clocks `V(A)` and `V(B)` can be compared: `V(A) ≤ V(B)` if every entry in V(A) is ≤ the corresponding entry in V(B); `V(A) < V(B)` if V(A) ≤ V(B) and the two vectors are not equal; and V(A) and V(B) are **concurrent** if neither is less than the other. Vector clocks have the property that `A → B` if and only if `V(A) < V(B)` — they provide a complete characterisation of causality.

In practical systems, vector clocks are expensive: each clock requires O(n) storage and O(n) to compare, where n is the number of nodes. In a system with thousands or millions of nodes, the cost is prohibitive. **Dotted version vectors** and **interval tree clocks** are more efficient variants that track causality in peer-to-peer systems where the node set is dynamic. Most practical distributed databases — Riak, Amazon DynamoDB — use some form of version vector to detect conflicts between concurrent writes (Lecture 5).

**Hybrid logical clocks** (HLCs), introduced by Kulkarni et al. in 2014, combine the best of physical and logical clocks. An HLC maintains a pair `(physical_time, logical_counter)`. The physical component is the node's last known NTP-synchronised time; the logical component is incremented whenever the clock would otherwise go backward or stagnate. HLCs are bounded to within the NTP uncertainty of physical time — they provide the causal safety of vector clocks while being cheap to store (just two integers), human-readable (close to wall-clock time), and suitable for use in distributed databases and transaction protocols. CockroachDB uses HLCs as its primary timestamp mechanism.

## Lecture 5: Consensus — Paxos, Raft, and the Challenge of Agreement

Consensus is the most fundamental and most difficult problem in distributed computing. The problem is simple to state: a set of nodes must agree on a single value (a decision) despite the possibility of failures — node crashes, message loss, network partitions — and despite the fact that no node has complete information about the system state. The ability to reach consensus is the foundational building block on which all other distributed coordination mechanisms are built: leader election, atomic broadcast, distributed transactions, configuration management, and state machine replication all depend on a consensus protocol.

The **FLP impossibility result** (Fischer, Lynch, and Paterson, 1985) states that in an asynchronous distributed system where messages can be delayed arbitrarily and at least one node may crash, there is no deterministic algorithm that guarantees consensus in finite time. The proof is one of the most elegant and devastating in computer science: it shows that any algorithm that is supposed to solve consensus can be forced into an infinite sequence of state transitions in which the nodes never settle on a decision — because the delay of messages can mimic the behaviour of a crashed node, and the algorithm can never tell the difference. The FLP result does *not* mean that consensus is impossible in practice — it means that a practical consensus protocol must either (a) assume synchrony (bounded message delays) some of the time, or (b) use randomness (probabilistic termination), or (c) rely on failure detectors that may be inaccurate but are eventually correct.

**Paxos**, developed by Leslie Lamport and first presented in 1989 (though the famous "Part-Time Parliament" paper was not published until 1998), was the first practical consensus protocol. Lamport cast the protocol as a fictional parliament on the Greek island of Paxos, where legislators (nodes) passed decrees (decisions) despite absenteeism (failures) and lost records (message loss). The protocol works through a series of **phases**:

**Phase 1 (Prepare)**: A node that wishes to propose a value (the *proposer*) sends a `Prepare(n)` message to all *acceptors*, where n is a unique, monotonically increasing proposal number. Each acceptor responds with a promise: it will never accept any proposal with a number less than n, and it returns the highest-numbered proposal (if any) that it has already accepted.

**Phase 2 (Accept)**: If the proposer receives promises from a majority of acceptors, it sends an `Accept(n, v)` message to all acceptors, where v is the proposed value — either the value associated with the highest-numbered proposal among the promises received (if any), or the proposer's own value (if none). Each acceptor accepts the proposal unless it has already promised to accept a higher-numbered one.

If a majority of acceptors accept, consensus is reached: the value v is decided. Additional phases (**Phase 3 — Learn**) propagate the decision to all *learners*.

Paxos is famously difficult to understand and implement correctly. Lamport himself acknowledged this, noting that the number of implementations of Paxos that were "production-quality" could be counted on one hand for many years. The protocol's complexity arises from the fact that it was designed to be *optimal* — the first fast fault-tolerant consensus protocol, with minimal message rounds in the common case — and optimal algorithms are often intricate.

**Raft** (Ongaro and Ousterhout, 2014) was designed explicitly to be more understandable than Paxos while providing the same guarantees. Raft decomposes the consensus problem into three relatively independent sub-problems: **leader election**, **log replication**, and **safety**. An introduction.txt accompanying the paper described the design goal: "Raft is a consensus algorithm for managing a replicated log. It produces a result equivalent to Paxos, but is structured differently than Paxos. Raft's designers put a premium on understandability."

The architecture of Raft is built around a **leader**: at any time, one node is the leader, and all log entries flow from the leader to the followers. Clients send all requests to the leader, which appends them to its local log and then issues `AppendEntries` RPCs to the followers. When a majority of followers acknowledge that an entry has been replicated, the leader commits the entry and notifies the followers. If the leader fails, any follower can initiate an election — it increments its **term number** (a monotonically increasing logical clock), votes for itself, and requests votes from other nodes. A candidate becomes the new leader if it receives votes from a majority of the cluster.

Raft's **safety** property — that a leader can never overwrite an entry that has been committed in a previous term — is enforced through a **restriction on voting**: a node will not vote for a candidate whose log is less complete than its own (the candidate's log must be at least as up-to-date as the voter's). This ensures that the leader always has the most complete log, and therefore cannot be forced to overwrite entries that have been committed. The log entries themselves are replicated according to the standard state-machine replication pattern: each entry is executed deterministically on each node, so all nodes converge to the same state.

Raft has become the consensus protocol of choice for new distributed systems in the 2020s and 2030s. It is implemented in etcd (the coordination layer for Kubernetes), Consul (Hashicorp), TiKV (a distributed transactional key-value store), YugabyteDB, CockroachDB's consensus layer, and many others. The University of Yggdrasil's Distributed Systems Lab includes a full educational implementation of Raft in Rust (the *Yggdrasil Raft* library), which students use to experiment with leader elections, log replication, and failure scenarios in the Pi cluster.

## Lecture 6: Replication Strategies and Consistency Models

Replication — maintaining copies of the same data on multiple nodes — is the foundation of availability and durability in distributed systems. If data exists on only one node, the failure of that node means permanent data loss. If data exists on multiple nodes, the failure of any single node is survivable — clients can be redirected to a surviving replica. But replication introduces the problem of **consistency**: if multiple replicas can be updated independently, how do we ensure that all replicas eventually have the same data, and what do clients see while replicas are converging?

**Synchronous replication** (also called *strong consistency* or *eager replication*) requires that a write operation does not complete until all replicas have acknowledged it. The advantage is simple: all replicas always contain the same data — clients can read from any replica and get the same result. The disadvantage is performance: synchronous replication magnifies latency (the write is not complete until the slowest replica has responded), and it reduces availability (if any replica is unreachable, the write cannot complete). In practice, synchronous replication is typically implemented as **majority quorum**: the write completes when a majority of replicas have acknowledged it (not all replicas), and the read retrieves the data from a majority of replicas. Read quorum + write quorum > N (the total number of replicas) ensures that read and write quorums overlap, so a read is guaranteed to see the most recent write.

**Asynchronous replication** (also called *lazy replication*) allows the write to complete after acknowledging on a single node (typically the leader), with the write propagated to other replicas in the background. The advantage is low latency and high availability — writes never block. The disadvantage is consistency: after a write, clients may read from a replica that has not yet received the update, returning stale data. Asynchronous replication is used by most large-scale web services (Facebook, Instagram, Twitter) because low latency is critical for user experience, and most of the data (user profiles, posts, comments) can tolerate transient inconsistency.

**Consistency models** describe what guarantees a replicated data store provides to concurrent readers and writers. The strongest model is **linearisability** (also called *atomic consistency* or *strong consistency*): every operation appears to occur atomically at a single point in time, and all operations are totally ordered — the system behaves as if there were a single, instantaneously consistent copy of the data. Linearisability is what most programmers implicitly assume when they use a variable in a sequential program, and it is what most relational databases provide (with Serializable isolation). Linearisability is expensive in a distributed setting: it requires coordination, quorums, and synchrony, and it is fundamentally incompatible with availability during partitions (the CAP theorem).

**Sequential consistency** (Lamport, 1979) is weaker than linearisability: it requires that the operations of each individual client are seen in the order they were issued by that client, but the operations of different clients can be interleaved arbitrarily. Sequential consistency is a natural model for multiprocessor systems with shared memory.

**Causal consistency** (Hutto and Ahamad, 1990) is weaker still: it requires only that causally related operations are ordered — writes that are causally dependent on other writes must be seen in the order of the causal chain — but concurrent writes (unrelated by causality) can be seen in any order. Causal consistency is the minimum consistency model that respects the "happened-before" relation. It is significantly cheaper than linearisability (no global coordination required) and is the target consistency level for many geo-distributed systems.

**Eventual consistency** is the weakest consistency model: it guarantees only that, if no writes occur for a sufficiently long period, all replicas will eventually converge to the same state. Until convergence, reads may return any value — including stale values, missing values, or values that were never written (if concurrent updates conflict). Eventual consistency is the default in many NoSQL databases (DynamoDB, Cassandra, Riak) and in DNS — it is cheap, available, and partition-tolerant, but places a heavy burden on the application to handle inconsistency.

**Conflict-free Replicated Data Types (CRDTs)**, developed by Shapiro, Preguiça, Baquero, and Zawirski in 2011, are data structures that automatically resolve concurrent updates without conflict resolution — they are designed so that operations commute: given the same set of operations applied in any order, all replicas reach the same state. The two main classes of CRDTs are **state-based** (convergent replicated data types, CvRDTs) — replicas periodically exchange their full state, and the merge function is commutative, associative, and idempotent — and **operation-based** (commutative replicated data types, CmRDTs) — operations are broadcast to all replicas, and any two concurrent operations commute. Practical CRDTs include the **G-Counter** (grow-only counter: each node has its own entry, and the total is the sum of all entries — increments commute), the **PN-Counter** (positive-negative counter: a pair of G-counters, one for increments and one for decrements), the **G-Set** (grow-only set: elements can be added but never removed — add operations commute), the **2P-Set** (two-phase set: elements can be added and removed, but a removed element cannot be re-added — uses a tombstone set), and the **LWW-Register** (last-writer-wins register: each write is timestamped, and the write with the largest timestamp wins — the system must agree on a total order of timestamps). CRDTs are the foundation of collaborative editing systems (Google Docs offline mode, the Yjs framework for collaborative web applications), distributed counters (Redis CRDT-based replication), and peer-to-peer databases (Automerge, Replicache).

## Lecture 7: Distributed Storage — From Key-Value Stores to NewSQL and Beyond

Distributed storage systems are the backbone of modern cloud infrastructure. They range from simple key-value stores (Amazon DynamoDB, Redis Cluster) through wide-column stores (Cassandra, HBase) and document databases (MongoDB, Couchbase) to NewSQL systems that provide full SQL semantics on a distributed architecture (Google Spanner, CockroachDB, YugabyteDB). Each design makes different trade-offs along the dimensions of consistency, availability, partition tolerance, performance, and query expressiveness.

**Amazon Dynamo** (DeCandia et al., 2007) was a landmark system that introduced the **consistent hashing** based architecture that underlies many later distributed key-value stores. Consistent hashing assigns each data item to a node by hashing the item's key and mapping the hash to a position on a ring of hash values; each node is responsible for the segment of the ring between itself and its predecessor. When nodes join or leave the ring, only the immediate neighbours are affected — only O(1/N) of the keys need to be moved (where N is the number of nodes), compared to O(N) in traditional modulo-based partitioning. Dynamo provides **eventual consistency** with **quorum-based reads and writes**: each data item is replicated to a configurable number of *preference list* nodes; reads and writes target a configurable quorum size; and conflicts are resolved using **vector clocks** (the system stores multiple versions of an item; client applications resolve conflicts at read time). Dynamo also introduced the **sloppy quorum** and **hinted handoff** mechanisms: if a node is temporarily unavailable, the write is sent to an alternative node, which stores it as a "hint" and forwards it to the correct node when it comes back online — increasing availability during transient failures.

**Google Spanner** (Corbett et al., 2012) took the opposite approach: a globally distributed database that provides **external consistency** (a correctness guarantee stronger than linearisability) across data centres on different continents. Spanner achieves this remarkable feat through two technologies: (1) **TrueTime**, a clock synchronisation system based on GPS receivers and atomic clocks installed in each data centre, which exposes a `[earliest, latest]` interval indicating the uncertainty bound on the current time; and (2) a variant of the Paxos consensus protocol that uses TrueTime intervals to timestamp transactions so that they are ordered consistently with causality. Spanner provides SQL semantics with schema, secondary indexes, and ACID transactions — it is a relational database that happens to be distributed across the planet. The key insight is that TrueTime reduces clock uncertainty to a known bound (typically 1–7 ms), and the database can use this bound to enforce serialisability without the performance cost of traditional distributed locking protocols. Spanner powers Google's advertising infrastructure, Google Play, and Gmail — systems that require both global scale and strong consistency.

**CockroachDB**, inspired by Spanner but designed for the open-source ecosystem, provides similar guarantees (external consistency, SQL, ACID transactions) without requiring specialised hardware. Instead of GPS clocks, CockroachDB uses **hybrid logical clocks** (Lecture 4) — each node's clock is synchronised via NTP, and the logical component ensures that clock drift is compensated for. Transactions use a **two-phase commit with parallel commits** optimisation (introduced in CockroachDB 20.1) and a **layered storage engine** (Pebble, a Go-based LSM tree) that provides fast point reads and range scans. CockroachDB's architecture is a **multi-active** design — every node can serve reads and writes, not just a leader — achieved by range-based sharding (each table is split into *ranges*, each range is an independently replicated consensus group using the Raft protocol) and a distributed transaction coordinator that routes requests to the appropriate range leader.

The future of distributed storage in 2040 is shaped by several trends: **disaggregated storage** (compute and storage are separated — compute nodes that fail lose only their computation, not the data; storage nodes are optimised for density and durability, not CPU performance); **NVMe-over-Fabrics** (non-volatile memory is shared at microsecond latencies across a network, blurring the boundary between local and remote storage); **programmable storage** (storage nodes can execute user-defined functions — filtering, aggregation, machine learning inference — in the storage layer, reducing data movement); and **conflict-free data types** embedded directly into the storage engine (so that the database itself is a CRDT, not an add-on layer). The University of Yggdrasil's *YggNAS* research project is exploring a distributed storage architecture based on disaggregated NVMe-over-Fabric and programmable RISC-V storage controllers.

## Lecture 8: Distributed Computation — MapReduce, Stream Processing, and the Lambda Architecture

Distributed storage is only half the picture — the other half is distributed computation: the ability to process massive datasets that are too large to fit on a single machine, or to process data in real time as it arrives. The last two decades have seen an explosion of distributed computation frameworks, from the batch-oriented MapReduce paradigm through real-time stream processing to the unification of batch and streaming in the "streaming-first" model.

**MapReduce** (Dean and Ghemawat, 2004) is a programming model and an associated implementation for processing large datasets in a distributed cluster. The programmer specifies a **map** function — a pure function that processes a single key-value pair and emits zero or more intermediate key-value pairs — and a **reduce** function — a function that merges all intermediate values associated with the same key. The system handles: partitioning the input data across the cluster, scheduling tasks on available nodes, handling node failures (re-executing failed tasks), and shuffling intermediate data from mappers to reducers over the network. The elegance of MapReduce is that any computation that can be expressed as a map followed by a reduce can be executed in parallel across an arbitrarily large cluster — the programmer does not need to think about parallelism, distribution, or fault tolerance. MapReduce was used at Google for most of the company's large-scale data processing for nearly a decade, including the indexing pipeline for Google Search, the PageRank algorithm, and the statistical machine translation system.

**Apache Hadoop** (2006) is the open-source implementation of MapReduce. Its architecture includes HDFS (Hadoop Distributed File System — a distributed filesystem that stores data in 128 MB blocks, replicated across three nodes by default), YARN (Yet Another Resource Negotiator — the cluster resource management layer that schedules MapReduce jobs alongside other workloads), and the MapReduce execution engine. Hadoop became the de facto standard for large-scale data processing in the 2010s, powering data analytics pipelines at virtually every large organisation.

**Apache Spark** (Zaharia et al., 2010) improved on MapReduce by introducing **in-memory computation** and **resilient distributed datasets (RDDs)**. In MapReduce, intermediate data is written to disk between the map and reduce phases — which is expensive for iterative algorithms (machine learning, graph processing) that require multiple passes over the data. Spark keeps intermediate data in memory, reducing disk I/O by orders of magnitude. RDDs are fault-tolerant collections that can be partitioned across nodes: each RDD records its *lineage* — the sequence of operations that created it — and if a partition is lost, it can be recomputed by replaying the lineage. Spark also introduced **DataFrames** — a higher-level API inspired by data frames in R and Python (pandas) — and **Spark SQL** — a query planner and execution engine that optimises SQL queries across the distributed execution environment.

**Stream processing** is the paradigm for processing data as it arrives, not in batches. The fundamental insight of stream processing — captured by the **Lambda Architecture** (Nathan Marz, 2011) — is that real-time and batch processing serve different but complementary roles: the batch layer recomputes the entire dataset periodically, providing accuracy; the speed layer processes data in real time, providing low latency; and the serving layer merges results from both. The Lambda Architecture was influential but had significant drawbacks: it required maintaining two separate codebases (batch and streaming) that must produce consistent results, and the operational complexity was high.

The **Kappa Architecture** (Jay Kreps, 2014) simplified this picture by treating all data as a stream: the entire dataset is stored in a distributed log (Kafka), and both real-time and historical processing use the same streaming engine — historical processing is simply replaying the log from the beginning. The Kappa Architecture eliminates dual-codebase complexity: a single streaming application can process both real-time and batch workloads. Apache Flink, Apache Kafka Streams, and Apache Samza are the primary Kappa Architecture frameworks. Flink's programming model is based on **stateful stream processing**: operators can maintain state (keyed state, window state, operator state) that is persisted, checkpointed, and recovered automatically — the system provides exactly-once semantics for stateful computations, even in the presence of failures.

**Differential dataflow** (McSherry et al., 2013) and **timely dataflow** (the Naiad system, Murray et al., 2013) represent the cutting edge of distributed computation in 2040. These systems combine batch and stream processing at a more fundamental level: they track data *changes* rather than data *snapshots*, so a computation on a large dataset can be incrementally updated when new data arrives — the system recomputes only the affected outputs, not the entire result. Timely dataflow is the foundation of Materialize, a streaming SQL database that provides materialised views that are updated in real time as the underlying data changes. The Yggdrasil Distributed Computing Lab runs experiments on timely dataflow in the Pi cluster, measuring the performance of incremental view maintenance across different cluster configurations and network topologies.

## Lecture 9: Distributed Coordination — Locks, Barriers, and Configuration Management

Distributed coordination is the set of mechanisms that allow distributed systems to synchronise their behaviour — to elect a leader, to ensure that only one node performs a critical operation at a time, to agree on a configuration, and to detect failures. While consensus protocols (Raft, Paxos) provide the fundamental building block for agreement, practical coordination systems provide higher-level primitives that application developers can use without implementing consensus from scratch.

**Apache ZooKeeper** (Hunt et al., 2010) is the original and most widely deployed distributed coordination service. ZooKeeper provides a hierarchical namespace (like a filesystem) with small data nodes (znodes) that can store configuration information, metadata, and synchronisation primitives. Clients can read and write znodes, and they can set **watches** — one-time triggers that fire when a znode changes, notifying the watcher asynchronously. ZooKeeper's API supports: **sequential znodes** (znodes whose names include a monotonically increasing sequence number — used to implement distributed queues and leader election); **ephemeral znodes** (znodes that are automatically deleted when the client session ends — used to detect client failures); and **recipes** — higher-level distributed data structures built on these primitives, including distributed locks, barriers, leader election, and group membership. ZooKeeper uses a variant of Paxos (Zab, the ZooKeeper Atomic Broadcast protocol) to replicate state across an ensemble of servers.

**etcd** (2013) is a simpler, modern alternative to ZooKeeper, built around the Raft consensus protocol (Lecture 5). etcd provides a key-value store with a gRPC API, watch support (long-lived watches that stream changes to clients), and a lease mechanism (similar to ephemeral znodes — the lease is kept alive by periodic heartbeats, and the associated keys are deleted if the lease expires). etcd is the coordination layer for Kubernetes: it stores the entire cluster state (pods, services, configurations, secrets) and provides the consistency guarantees that Kubernetes requires. The flexibility of etcd's watch mechanism — clients can watch a single key, a range of keys, or a prefix — makes it suitable for service discovery, configuration management, and distributed task scheduling.

**Leader election** is one of the most common coordination patterns. In the simplest approach — using ZooKeeper sequential ephemeral znodes — each candidate creates an ephemeral sequential znode under a leader election path; the candidate with the smallest sequence number becomes the leader, and all others watch the preceding znode to detect when the leader fails. In etcd, leader election is typically implemented using the `concurrency` package: candidates attempt to create a key with the same name (using `put` with the `prevValue` condition or using a `campaign` operation); the first candidate to succeed becomes the leader, and the others watch the key to detect leadership changes. In both cases, the watch mechanism is critical: it allows non-leaders to react immediately when the leader fails, rather than polling.

**Distributed locks** — ensuring that only one node executes a critical section at a time — are more subtle than leader election. The challenge is that a lock acquired by a node that subsequently fails or becomes unresponsive will never be released, causing the system to deadlock. ZooKeeper and etcd both solve this through **leases** (also called *lock leases*): the lock-holding node must periodically renew its lease by sending heartbeats; if the lease expires, the lock is automatically released, and another node can acquire it. The lease duration must be chosen carefully — too short, and a slow (but not failed) node may lose its lock prematurely; too long, and the system may be blocked waiting for a truly failed node's lease to expire. Chubby, Google's distributed lock service (Burrows, 2006), used a default lease time of 10 seconds.

**Configuration management** — the ability to change system configuration dynamically without restarting — is another essential coordination function. In a distributed system, configuration changes (adding a node, changing a timeout, updating the replication factor) must be applied consistently across all nodes, and the system must handle the transition between configurations gracefully. **Rolling updates** — updating one node at a time — are the most common strategy, but they require coordination: before updating node N, the system must ensure that the cluster can tolerate the temporary removal of N from the replica set. **Two-phase configuration changes** (described by Ongaro for Raft) add the new configuration atomically: the cluster first agrees to enter a *joint consensus* phase where both old and new configurations are active, then transitions to the new configuration alone. This ensures that the cluster can survive failures during the configuration change.

The Yggdrasil Distributed Systems Lab uses the *Yggdrasil Coordination Service* — a custom coordination layer based on the Raft protocol, implemented in Rust with a gRPC API and WebAssembly-based server-side filters. The service is deployed across the 32-node Pi cluster and is used as the coordination backbone for student projects: distributed game servers, collaborative editing applications, and peer-to-peer data sharing tools.

## Lecture 10: Byzantine Fault Tolerance and Permissioned Consensus

The consensus protocols we have studied (Paxos, Raft) assume **crash-fault** behaviour: nodes may stop responding, but they never deviate from the protocol — they never send false messages, collude with other nodes, or attempt to subvert the system. In many environments—financial systems, blockchains, military command-and-control, or any system that must operate in the presence of malicious attackers — we must assume a stronger failure model: **Byzantine** failures, in which nodes may behave arbitrarily. A Byzantine node can send conflicting messages to different receivers, impersonate other nodes, replay old messages, collude with other Byzantine nodes, or simply crash. The **Byzantine Generals Problem** — first formulated by Lamport, Shostak, and Pease in 1982 — imagines several divisions of the Byzantine army camped outside an enemy city, each commanded by a general who must decide whether to attack or retreat. They communicate by messenger, but one or more generals may be traitors, actively working to prevent the loyal generals from reaching agreement. The question: can the loyal generals reach a consensus on a common plan, despite the traitors?

The answer, established in the same 1982 paper, is that in a system with `f` Byzantine nodes, consensus is possible if and only if `n ≥ 3f + 1` — the total number of nodes must be at least three times the number of Byzantine nodes plus one. Intuitively, each loyal node needs to be able to out-vote the Byzantine nodes even in the worst case (when all Byzantine nodes behave maliciously), and the `3f + 1` bound ensures that the loyal nodes form a strict majority of non-Byzantine nodes. The original **oral messages** protocol — in which messages are unsigned — requires at least `3f + 1` nodes and `f + 1` rounds of message exchange. A more recent protocol variant, **signed messages** (using digital signatures), resolves disagreement more efficiently by making it impossible for a node to forge another node's message.

**Practical Byzantine Fault Tolerance (PBFT)**, developed by Miguel Castro and Barbara Liskov in 1999, was the first practical BFT protocol — previous protocols were either too slow (exponential message complexity) or required too many nodes. PBFT provides state-machine replication in the Byzantine model with `O(n²)` message complexity (each node sends messages to all other nodes in the worst case), which is acceptable for small to medium-sized systems (dozens to hundreds of nodes). PBFT operates in three phases: **pre-prepare** (the primary — a designated leader — proposes a request); **prepare** (nodes broadcast that they have received the request and intend to commit it — this ensures that all non-faulty nodes agree on the request's order); and **commit** (nodes broadcast that they have seen a prepare quorum — this ensures that the request is committed even if the primary fails). PBFT was used in several permissioned blockchain systems (Hyperledger Fabric v0.6, certain early versions of Ripple) and is still studied as the baseline for Byzantine consensus.

**Byzantine Fault Tolerant versions of Raft and Paxos**, such as BFT-SMaRt (Bessani, Sousa, and Alchieri, 2014), adapt the crash-fault protocols to the Byzantine model. These protocols use **threshold signatures** (a signature that requires cooperation from a subset of nodes to produce or verify) and **view changes** (replacing a faulty primary) that are more complex than in crash-fault systems because a faulty primary can actively try to prevent a successful view change. BFT-SMaRt is used in several European banking systems and was adopted by the Linux Foundation's Hyperledger project.

**Proof-of-Work (PoW) consensus** — the mechanism underlying Bitcoin — takes a fundamentally different approach to Byzantine consensus. Instead of bounding the number of Byzantine nodes (`3f+1`), PoW assumes an *open membership* system: any node can join or leave at any time, and there is no limit to the number of Byzantine nodes. Security is achieved through **economic incentives**: the cost of attacking the system (electricity, hardware) must exceed the expected reward. A node that proposes a valid block (a block whose hash is below a target difficulty) is rewarded with newly created coins; nodes that propose invalid blocks lose the cost of the computation. The system is secure as long as the attacker controls less than 50% of the total computational power (the hash rate). **Proof-of-Stake (PoS)**, used by Ethereum after its 2022 Merge, replaces computation with economic stake: nodes deposit cryptocurrency as collateral, and if they are caught behaving dishonestly, their stake is *slashed* (destroyed). PoS is more energy-efficient and provides faster finality than PoW, but it requires more complex protocol mechanisms to prevent *nothing-at-stake* attacks (validating on multiple forks simultaneously) and *long-range attacks* (rewriting history from early in the chain).

The 2040 landscape of Byzantine consensus is dominated by **practical Byzantine infrastructure** in both permissioned and permissionless settings. In permissioned systems — financial consortia, supply chains, identity management — PBFT-style protocols with threshold signatures and hardware security modules (HSMs) provide high throughput and low latency. In permissionless systems — public blockchains, decentralised AI marketplaces, decentralised identity — PoS consensus with modern refinements (finality gadgets, attestation committees, and sub-epochs) provides throughput exceeding 100,000 transactions per second. The Yggdrasil Distributed Systems Lab studies **layered BFT** protocols that combine PBFT's strong safety guarantees with PoS's open membership: a small, frequently rotated *committee* runs PBFT at high speed, while a larger *validator set* selects and rotates the committee members through PoS voting.

## Lecture 11: Observability, Debugging, and Testing Distributed Systems

Distributed systems are notoriously difficult to debug. A request may traverse dozens of nodes, each of which may add latency, fail, or behave unexpectedly — and the system's behaviour emerges from the interactions between these nodes, not from the individual behaviour of any single node. Observability — the ability to understand what the system is doing, and why — is not a luxury in distributed systems; it is a fundamental requirement for operating them reliably.

**The three pillars of observability** — **logs**, **metrics**, and **traces** — provide complementary views of system behaviour. **Logs** are structured or unstructured records of discrete events: a request arrived, a database write completed, a connection was closed. In a distributed system, logs must include **correlation IDs** (or trace IDs) — a unique identifier that is propagated through all nodes that handle a request — so that the logs from different nodes can be associated with the same request. **Metrics** are numeric aggregations over time: request latency (p50, p99, p999), error rate, throughput, CPU utilisation, memory usage. Metrics are aggregated and stored in time-series databases (Prometheus, InfluxDB, TimescaleDB) and are the primary tool for detecting anomalies and understanding long-term trends. **Traces** are records of the path a request takes through the system: each *span* represents a unit of work (an RPC call, a database query, a cache lookup), and spans are connected by parent-child relationships that form a trace — a tree of spans that shows exactly how long each operation took and which operations depended on which.

**Distributed tracing** is the most important and most challenging of the three pillars. The standard is the **OpenTelemetry** framework (merged from OpenTracing and OpenCensus in 2019), which provides APIs for instrumenting code to produce traces, a protocol for transmitting trace data (gRPC or HTTP), and a specification for the data model (the *trace context* — trace ID, span ID, and propagation format). OpenTelemetry has become the universal standard for observability instrumentation: virtually every major framework, library, and runtime provides OpenTelemetry-compatible instrumentation out of the box, and observability backends (Jaeger, Zipkin, Grafana Tempo, Honeycomb) accept OpenTelemetry data natively. In 2040, the University of Yggdrasil's distributed systems infrastructure was one of the early adopters of OpenTelemetry's specification for the binary propagation format.

**Testing distributed systems** requires techniques that go beyond unit and integration testing. **Simulation-based testing** — pioneered by the *FoundationDB* team and later by *Jepsen* (Kyle Kingsbury) — tests the system under controlled failure conditions: network partitions, node crashes, clock skew, message reordering, and Byzantine behaviour. Jepsen runs a workload against a distributed system while introducing failures, then checks whether the system's behaviour is consistent with its specification — whether it has lost data, violated consistency guarantees, or produced incorrect results. Jepsen's annual reports on distributed systems (Cassandra, MongoDB, etcd, ZooKeeper, Elasticsearch) have been instrumental in identifying correctness bugs in production systems and have raised the bar for distributed systems quality assurance.

**Formal verification** is increasingly applied to distributed protocols. **TLA+** (Lamport, 1999) is a formal specification language for concurrent and distributed systems — a TLA+ specification describes all possible behaviours of the system, and the TLC model checker explores these behaviours exhaustively (for small state spaces) to verify that dangerous states (deadlock, data loss, inconsistency) are never reached. TLA+ was used to verify the Raft consensus protocol (Ongaro, 2014 — a formal TLA+ specification of Raft was written as part of the PhD thesis), the Amazon Web Services DynamoDB consistency model, and the Hyperledger Fabric blockchain protocol. **Ivory** and **P** are domain-specific languages for writing distributed protocols that can be compiled to both executable code and formal models — the code and the model are guaranteed to be consistent. In the Yggdrasil Distributed Systems Lab, students write distributed protocols in **P**, verify them with the **P** model checker, and deploy them directly to the Pi cluster — the same source code runs in the laboratory and in the verified model.

**Fault injection testing** — deliberately introducing failures into a running system — is the standard practice for validating production resilience. Netflix's **Chaos Monkey** (2011) was the first widely known fault injection tool: it randomly terminates instances in production to ensure that the system can survive instance failures. Chaos Monkey was later expanded into the **Simian Army** (a suite of tools that inject different kinds of failures — latency, network partitions, DNS failures, certificate expiration) and then into **Chaos Engineering** — the discipline of experimenting on a production system to build confidence in its ability to withstand turbulent conditions. Principles of Chaos Engineering include: (1) **minimise blast radius** — the failure injection should affect only a subset of users or traffic; (2) **run experiments continuously** — reliability is not a one-time fix but a continuous property; (3) **verify the steady state** — the experiment should confirm that the system maintains its steady-state behaviour (latency, throughput, error rate) despite the injected failure. The Yggdrasil Lab runs a Chaos Engineering experiment on the Pi cluster every academic day: one node is randomly killed, one network link is partitioned, and one process is given an artificial delay — and the student operators must detect, diagnose, and recover from each failure.

## Lecture 12: The Future — Edge Computing, Serverless, and the Distributed Web

Distributed systems in 2040 are shaped by three dominant trends: the move toward **edge computing** (computation at the network edge, close to data sources and users); the maturation of **serverless computing** (execution environments that abstract away servers entirely); and the emergence of a **distributed web** (decentralised alternatives to centralised cloud services, built on peer-to-peer protocols and blockchain infrastructure). These trends are not independent — they reinforce each other, and their convergence defines the distributed systems research agenda for the next decade.

**Edge computing** distributes computation and data storage across a hierarchy of tiers: the **device tier** (sensors, actuators, smartphones, IoT endpoints — constrained in power, memory, and connectivity); the **edge tier** (gateways, base stations, local servers — within 1–10 ms of the device tier, typically with modest resources and intermittent connectivity to the cloud); and the **cloud tier** (large data centres with abundant resources but high latency — 10–100+ ms from the device tier). The key observation is that many applications — autonomous vehicles, augmented reality, industrial automation, smart cities — require millisecond-level response times that are impossible if all processing must travel to the cloud. Edge computing pushes latency-critical computation to the edge tier, while non-latency-critical computation (data aggregation, model training, long-term storage) remains in the cloud. The fundamental distributed systems challenge of edge computing is **variable connectivity**: edge nodes may be offline for seconds, minutes, or hours (a car in a tunnel, a factory sensor during maintenance), and the system must operate correctly in disconnected, intermittent, and limited (DIL) network conditions — a pattern known as **offline-first** computing. Offline-first systems use CRDTs (Lecture 6) extensively for conflict resolution during synchronisation windows.

**Serverless computing** (also called Function-as-a-Service, FaaS) abstracts the compute node completely. The programmer writes a *function* — a single operation, potentially stateless — and the cloud provider executes it in response to an event (an HTTP request, a message on a queue, a file upload, a database change). The provider manages all infrastructure: scaling (functions are scaled from zero to thousands of instances based on demand), scheduling (functions are placed on available compute resources), and failure recovery (if a function invocation fails, it may be retried automatically). Serverless computing dramatically simplifies the programming model: the programmer does not think about servers, clusters, or load balancers — only about functions, events, and data. The challenges of serverless from a distributed systems perspective are **cold starts** (when a function is idle, its container is reclaimed; when a new request arrives, a new container must be started and initialised — this can add 100–1,000 ms of latency), **state management** (functions are stateless by design, but many applications need state; the state must be stored in an external service — a key-value store, a database, a cache — which reintroduces the distributed systems challenges of replication, consistency, and latency), and **function orchestration** (complex workflows require many functions to be composed — executing them in sequence, parallel, or conditional patterns — which is itself a distributed coordination problem solved by workflow engines such as AWS Step Functions, Azure Durable Functions, or Temporal).

The **distributed web** (also called Web3 or the decentralised web) is an attempt to rebuild the Web's infrastructure on decentralised, peer-to-peer protocols, eliminating reliance on centralised cloud providers and single points of control. Key infrastructure components include: **IPFS** (InterPlanetary Filesystem) — a content-addressed, peer-to-peer filesystem in which files are identified by their cryptographic hash, not by their location on a specific server; **Filecoin** — a decentralised storage network that provides proof-of-replication and proof-of-spacetime to verify that storage providers are actually storing files; **Libp2p** — a modular network stack for peer-to-peer applications that provides transport, security, peer discovery, content routing, and NAT traversal; and **Ethereum** and **Cosmos** — smart-contract platforms that provide decentralised consensus (PoS) and a global state machine that is maintained by a distributed network of validators. The distributed web faces severe distributed systems challenges: **scalability** (a decentralised network with thousands of validators cannot achieve the throughput of a centralised system with a handful of nodes); **latency** (consensus across thousands of nodes requires multiple round-trips, adding seconds or minutes to finality); **usability** (key management, transaction fees, and the complexity of interacting with smart contracts remain barriers for non-technical users); and **governance** (how are protocol upgrades decided in a system that has no central authority?).

The University of Yggdrasil is at the forefront of distributed web research. The **Yggdrasil Distributed Protocol Laboratory** (a collaboration between the Computer Science and the AI OS Design programs) has developed a **serverless edge computing framework** — *Yggdrasil Edge Functions* — that applies the serverless programming model to the edge hierarchy: functions can be deployed to any tier (device, edge, cloud) based on latency and resource requirements, and the runtime automatically manages migration between tiers as network conditions change. The **Yggdrasil Distributed Web Initiative** (YDWI) runs a small-scale distributed web testbed on the 32-node Pi cluster, running IPFS, Libp2p, and an Ethereum-consensus-compatible PoS validator client. Students in the course have the opportunity to deploy their own distributed applications — chat systems, collaborative editors, decentralised identity solutions — on this testbed and evaluate their performance under realistic conditions (simulated network partitions, node churn, and adversarial behaviour).

### Sample Essay Questions (Select 4 of 8)

1. **The CAP theorem in the real world.** Discuss a real distributed system of your choice (e.g., Amazon DynamoDB, Google Spanner, Cassandra, CockroachDB). Analyse its design through the lens of the CAP theorem: which of C, A, and P does it prioritise, how does it handle partitions, and what consistency guarantees does it actually provide to clients? Do you agree with the design choices made by the system's architects?

2. **Raft vs. Paxos: a comparative analysis.** Compare Raft and Paxos in terms of understandability, performance under normal conditions, performance under failure conditions, and suitability for different deployment scenarios. Under what circumstances would you choose one over the other? Is Raft always the better choice because it is more understandable, or are there scenarios where Paxos's generality is essential?

3. **CRDTs in collaborative applications.** CRDTs enable offline-first collaboration, but they have limitations (tombstone-based deletion in 2P-Sets, lack of support for arrays with arbitrary insertion/deletion without sequence-based identifiers). Design a CRDT-based collaborative text editor and discuss the trade-offs involved. How would you handle undo operations (which are inherently non-commutative) in a CRDT framework?

4. **Distributed consensus with Byzantine actors.** The PBFT protocol requires `3f+1` nodes to tolerate `f` Byzantine nodes. Explain why `2f+1` nodes are insufficient, using a concrete example of a Byzantine attack that would succeed in a `2f+1` system. Then discuss how Proof-of-Stake consensus relaxes this bound — does it provide equivalent safety, and if not, what additional assumptions does it rely on?

5. **Design a distributed key-value store.** You are building a distributed key-value store for an IoT edge computing deployment: 100 edge nodes, each with limited storage (32 GB) and intermittent connectivity (nodes may be offline for up to 1 hour per day). Design the replication strategy, consistency model, and conflict resolution mechanism. Justify your design choices with reference to the theoretical frameworks covered in this course.

6. **Observability in microservices.** A distributed request in your microservice architecture traverses 15 services, 3 message queues, and 2 databases. Design an observability strategy (logs, metrics, traces) that would allow you to diagnose this class of "silent data loss" bugs. What specific instrumentation would you add, and how would you analyse the resulting data to find the bug?

7. **Time and causality in distributed databases.** CockroachDB uses hybrid logical clocks (HLCs) to provide external consistency without specialised hardware (unlike Spanner's TrueTime). Explain how HLCs work, how they compare to TrueTime in terms of guarantees and performance, and under what conditions HLC-based systems provide weaker guarantees than TrueTime-based systems. Design a scenario in which an HLC-based system would produce an incorrect result that a TrueTime-based system would correctly avoid.

8. **The future of distributed systems.** Choose one of the three trends from Lecture 12 (edge computing, serverless, or the distributed web) and argue for or against its long-term significance. What is the single most important unsolved distributed systems problem in that space, and how would you approach it? Your answer should demonstrate a thorough understanding of the theoretical foundations covered in this course.
